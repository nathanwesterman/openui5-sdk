Keyboard Usage of ARIA Role Mapped Controls

The JAWS screen reader offers list features that ease the app navigation by grouping and extracting all elements with similar behavior - for example, all headings or all tables. This leads to additional requirements when building apps with OpenUI5 controls.

Keyboard Usage of Role Mapped Controls

The applied role names define implicitly the keyboard usage,for example an element with role=button can be activated with SPACE and ENTER keys, navigation between controls with role=menu item is expected using Arrow keys and so on. The screen reader uses the role information to read the keyboard navigation hints.

Note
If you develop new SAPUI5 controls please note thefollowing:

  • Navigation with the JAWS Virtual Cursor cannot be the only option. Keyboard navigation has to be implemented as well.

  • All information about roles, states, and properties must be present at the keyboard focus position and be spoken on control focus

Additional requirements for some control types

JAWS offers lists that group certain types ot elements. These lists ease and speed up the navigation of the screen reader. Please note the following requirements for some control types.

  • Landmarks/Regions

    • Special Regions must be indicated and labelled (navigation, Page header, Main etc.) and support the landmark list feature (Jaws INSERT + CONTROL + R landmark list or R for direct navigation)

  • Headings

    • The headings in a SAPUI5 app must have either aria role=heading + aria-levelor use <H1 – H6> HTML controls (JAWS INSERT + F6 headings list or H for direct navigation)

    • The panel heading hierarchy must be logical (for example, nested panels must have higher hierarchy levels)

    • Headings must be referenced by the containers they belong to (using aria-labelledby = “HEADING_ID”)

  • Links

    • Links in the in a SAPUI5 app must support the link list feature (Jaws INSERT + F7 link list or L for direct navigation) by being true <a> controls or having ARIA role=link.

  • Form Controls

    • Form fields in the in a SAPUI5 app must support the form field list feature (Jaws INSERT + F5 form field list) and show up as Input, Radio Buttons and so on.

    • Form fields must be correctly labelled and their List indicators have to be distinctive

  • Lists

    • Lists have to implement a navigation concept that allows using both arrow keys to go through list item entities, but also focusing individual active sub-parts of a list item.

    • Lists and list items should always be identified using correctly nested <ul> and <li> markup or alternatively by ARIA list and list item roles.

    • Sometimes list items may need special roles (for example menu item).

    • For lists with n items, speech output should always be "item number of n items", no matter how many items are visible. If n is not known, speech output should always be "item number"

  • Tables

    • Data Tables must be coded as such (valid HTML) and must support the table list feature (Jaws INSERT + CTRL + T table list or T for direct navigation) .

    • Tables must have titles.

    • Layout Tables for presentational purposes must be coded as such (using ARIA role=”presentation”). Then they are not displayed in the Jaws table list.

    • Editable and active cells may require special interaction models. During navigation, all screen reader relevant information must be available at the focus location.

    • Tables have to properly associate and identify (even hierarchical) row/column headers as their cells for the headers act as labels for the table cells.

    • In case there are no visible headers, but information is organized in a table-like layout with individual “visual” columns/cells, respective row/column identifier as part of the cell info has to be given.

    • For tables with n rows, speech output should always be "row number of n rows", no matter how many rows are visible. If n is not known, speech output should always be "row number".
